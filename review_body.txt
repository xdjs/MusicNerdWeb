## Code Review: MCP Artist Link Tools

This is a well-scoped, well-documented release. The AsyncLocalStorage pattern for threading auth context through the MCP SDK singleton server is the right call, and the WRITABLE_LINK_COLUMNS whitelist in artistLinkService.ts is a meaningful security improvement over the old inline allowlist. Test coverage is thorough and follows project conventions.

---

### Issues

1. **delete_artist_link missing already-null check**: The design doc specifies that if the current value is already null the tool should return an error. But the handler calls clearArtistLink unconditionally after validating artistId and siteName. The operation is idempotent but produces a misleading audit log entry with oldValue: null and newValue: null. If this check was intentionally dropped, a comment explaining why would help.

2. **WRITABLE_LINK_COLUMNS drift from urlmap**: New platforms added to urlmap will not be writable via MCP tools until WRITABLE_LINK_COLUMNS in artistLinkService.ts is also updated. An agent using a newly-added valid siteName via delete_artist_link will get a confusing error from inside clearArtistLink. Consider whether urlmap validation alone is sufficient or add a comment documenting the sync requirement.

3. **mcp_audit_log.action unconstrained at DB level**: The TypeScript signature constrains action to set or delete, but the Postgres column accepts any string. A CHECK constraint in the migration would harden this.

### Minor observations

- **artist-detail.ts YouTube normalisation**: Phase 1A guarantees no @ prefix for new writes, so the strip only applies to pre-migration rows. A brief comment noting that would clarify whether it is permanent or removable after a data backfill.
- **approveUGC error message improvement**: Surfacing the underlying error in the throw makes whitelist rejections visible in logs. Nice.
- **removeArtistData wallet bug comment**: Good to have documented; out of scope here.
